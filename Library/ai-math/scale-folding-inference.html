Scale Folding and Weight Sliding

Inference latency is often dominated by small, element-wise operations that follow large matrix multiplications. Symbolic "Scale Folding" moves these constants into the weights themselves at compile-time, eliminating the need for extra kernels at runtime.

The Problem

The "Death by a Thousand Kernels"

In models like Batch-Normalized CNNs or Scaled Dot-Product Attention, we often see a pattern where a matrix multiplication is followed by a scalar multiplication:
Result = (X × W) × g (where g is a scale factor).

On a GPU, this usually launches two kernels: one for the MatMul and one for the element-wise multiplication by g. Even if the second kernel is fast, the launch overhead and memory round-trip time add up across dozens of layers.

Symbolic Insight

Moving the Constant (Scale Folding)

Because matrix multiplication is linear, we can "slide" the scalar g past the input X and into the weight matrix W:
(X × W) × g = X × (W × g)

In a symbolic engine, this is handled by **Weight Sliding** rules.

  • Compile-Time: We compute W' = W × g once.
  • Inference-Time: We only execute X × W'.

This "folds" the scaling operation into the existing matrix multiplication, reducing the total kernel count and improving latency without changing the mathematical result.

Why It Matters

Edge AI and Real-Time Systems

Scale folding is a mandatory optimization for Quantization and Batch Normalization Folding. It is one of the primary ways that compilers like TensorRT and OpenVINO achieve high performance on edge hardware, where every microsecond counts.
