Grouped-GEMM for MoE Efficiency

Mixture of Experts (MoE) architectures present a unique challenge: executing multiple different matrix multiplications on different subsets of data simultaneously. Grouped-GEMM is the symbolic solution that allows these irregular operations to be packed into a single, high-efficiency GPU call.

The Problem

Irregular Expert Assignment

In an MoE model, token 1 might go to Expert A, while token 2 goes to Expert B. This creates many small, uneven work packets:
Expert_A = MatMul(X_subset1, W_A)
Expert_B = MatMul(X_subset2, W_B)

If X_subset1 has 10 tokens and X_subset2 has 50 tokens, Standard Batching fails because the dimensions don't match. Launching individual kernels for every expert results in GPU Under-utilization (low occupancy).

Symbolic Insight

From Stacking to Grouping

While Stacking works for uniform inputs, **Grouped-GEMM** is required for the ragged, dynamic nature of MoE routing.

A symbolic engine identifies the "Collection of MatMuls" pattern:
List[MatMul(Xi, Wi)]

Instead of forcing a single uniform shape, the engine transforms the list into a Grouped-GEMM Descriptor. This descriptor tells the GPU to treat the expert weights as a single logical block, but with multiple offsets and sequence lengths.

  • Static Weights: W_A, W_B, ... are pre-arranged in memory.
  • Dynamic Indexing: The engine generates a map of which tokens belong to which expert block.
Why It Matters

Scaling to Thousands of Experts

Without Grouped-GEMM, MoE models are forced to use "Padding" to make all expert calls the same size, which wastes up to 50% of the compute power.

Symbolic optimization of Grouped-GEMMs is the core technology in high-performance MoE libraries like DeepSpeed-MoE and MegaBlocks. It allows models to scale the number of experts without increasing the latency of the dispatch loop.
