Transpose Hoisting for Fusion

The layout of tensors in memory—Row-Major vs Column-Major—determines the efficiency of GPU kernels. Symbolic "Transpose Hoisting" allows the engine to rearrange operations so that transposes are moved to the boundaries of the graph, enabling more fused operations in the "hot path."

The Problem

The Transpose Penalty

Transposing a large matrix in GPU memory is not a "free" operation; it involves a non-contiguous memory access pattern that can be quite slow. In many neural network implementations, transposes are sprinkled throughout the graph (e.g., after a Linear layer or before an Attention block).

Consider AT × BT. Executing this requires two explicit transposes before the multiplication can even begin.

Symbolic Insight

Hoisting the Transpose

A symbolic engine uses the identity:
AT × BT = (B × A)T

By "hoisting" the transpose outside the matrix multiplication:

  • The GPU can perform a high-speed MatMul on the original layouts of B and A.
  • A single transpose is performed on the result, or even better, deferred to the next layer's input.

This is especially powerful when combined with Relu Commutation, where Relu(AT) = (Relu(A))T, allowing the activation to fuse with the generating kernel before the transpose happens.

Why It Matters

CUDA Layout Optimization

In libraries like cuBLAS and CUTLASS, specific kernel variants (like GEMM_NT or GEMM_TN) are optimized for specific input layouts. Symbolic hoisting ensures that the graph is presented to the backend in a way that minimizes the need for explicit, slow memory shuffles.

// Original Expression MatMul(Transpose(A), Transpose(B))

// Rule Applied: TransposeMatMul (Inverse) MatMul(Transpose(?a), Transpose(?b)) -> Transpose(MatMul(?b, ?a))

// Commutation with Relu Relu(Transpose(X)) -> Transpose(Relu(X))
